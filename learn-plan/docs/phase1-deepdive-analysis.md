# Phase 1：深挖 + 分析

本文件定义 `/learn-plan` Phase 1 的执行要求。此阶段合并了原 clarification 和 research，目标是**同时做需求深挖和资料检索**。

## 1. 目标

不是直接写计划，而是完成两件事：
- 通过主题式多轮顾问访谈，把用户的真实需求收敛为结构化的学习画像
- 通过子 Agent 检索，搞清楚两件事：
  - **达标要求**：为达成用户目标，需要达到什么水平、掌握什么能力
  - **学习资料**：有什么可用的外部资料，够不够，缺什么

## 2. 执行流程

两部分可以交替进行，不必全部串行：

```text
  追问主题 1
    → 追问主题 2
    → 评估用户自备资料 → 发现缺口 → 启动资料检索
    → 追问主题 3
    → 资料检索结果回来 → 不够 → 告诉用户缺资料
    → 追问主题 4（资料条件相关）
    → ...
    → 全部必问主题完成 → 启动目的分析检索
    → 产出综合报告（目的分析 + 资料评估）
```

## 3. 深挖主题

每轮只聚焦一个主题，终端自然语言追问。

| 主题 ID | 追问方向 | 退出标准 |
|---|---|---|
| `learning_purpose` | 为什么学？考试的话追问范围/难度/真题/老师；求职的话追问目标公司/岗位/已看JD；项目的话追问交付物/验收标准 | 目的明确，场景具体（不能只是"考试高分""找工作"） |
| `success_criteria` | 怎样算学成？分数/排名/面试/offer/作品/证书 | 有可验证的成功标准 |
| `current_level` | 现有基础？相关经验？做过什么？ | 有具体描述，或显式 deferred 到 Phase 2 |
| `constraints` | 每周几天？每次多久？截止日？ | 时间/频率/截止日明确 |
| `teaching_preference` | 概念推导式（原理→应用）/ 情景案例式（故事→知识）/ 混合？默认情景案例式 | 偏好明确 |
| `practice_preference` | 题量偏好？题型偏好？反馈方式？ | 偏好明确或接受默认 |
| `materials` | 有没有自备资料（教材/真题/课程/视频/项目代码）？语言偏好？平台偏好？ | 已确认用户有什么、缺什么、想用什么语言/平台 |
| `assessment_budget` | 最多接受几轮起始测评？每轮最多几题？ | 预算已确认（默认 2 轮 × 10 题） |
| `non_goals` | 当前不学或后置的内容 | 明确排除项 |

### 追问示例

用户说"我想期末考高分"：
- 不要直接接受。应追问：
  - "是什么课程的期末考试？哪个学校的？"
  - "你知道考试范围吗？有没有往年真题可以参考？"
  - "考试形式是机考还是笔试？侧重理论还是编程？"
  - "你现在的目标分数是多少？目前大概什么水平？"

用户说"我想找 Python 后端的工作"：
- 继续追问：
  - "有具体的目标公司或岗位类型吗？看过哪些 JD？"
  - "你现在有没有项目经验？有没有简历可以给我看看？"
  - "有投递时间线吗？距离面试大概还有多久？"

### 追问规则

- 禁止用 AskUserQuestion / 选择题控件替代追问
- 不要一次抛出跨越多个主题的问卷
- 若用户回答模糊，继续同一主题，不跳题
- 若用户确实答不出，记录为 assumption 或 deferred，不伪造

## 4. 资料评估

先评估用户自备资料，再决定要不要搜。

### 自备资料评估

用户提供了资料时，评估：
- 覆盖度：能否覆盖目标所需的主要知识领域？
- 深度：内容深度是否匹配目标难度？
- 缺口：缺哪些章节/专题？缺练习吗？缺案例吗？

### 外部资料检索

- 当自备资料不满足需求时，派发多个子 Agent 并行检索
- 检索方向：课程文档、技术博客、GitHub 开源仓库、题库
- 检索的语言和平台偏好以用户确认为准

### 缺资料处理

**如果检索不到合适的公开资料，必须诚实告知用户**：
- 明确说"目前没找到合适的资料"以及具体缺什么
- 请用户自行提供或放宽检索条件
- 绝不虚构资料或编造内容

## 5. 目的分析检索

即使资料足够，仍需弄清楚"达成目标需要什么水平"。派发子 Agent 检索：

- **考试目标**：真题难度、高频考点、评分标准、历年分数线
- **求职目标**：岗位 JD 分析（2-3 个代表 JD）、常见面试题、技术栈要求
- **项目目标**：类似项目的技术选型、常见坑、交付标准

## 6. 产出：综合报告

Phase 1 最终产出是一份 HTML 格式的综合报告，包含：

1. **核心分析**：达成目标需要什么水平、掌握什么核心能力
2. **学习画像**：用户目的、基础、约束、偏好
3. **资料评估**：
   - 用户自备资料清单及评估（覆盖度/深度/缺口）
   - 检索到的补充资料清单，各资料角色（主线/辅助/可选）
   - 取舍说明（选什么、不选什么、为什么）
   - 缺的资料及原因
4. **来源证据**：各项结论的依据
5. **open risks**：未找到真题、资料覆盖不足、用户自述水平不可靠等
6. **Phase 2 预览**：如果需要诊断，测什么、大概几轮

报告是**中间产物**，用户可提出修改意见。

## 7. 用户可见输出

- 当前聚焦主题
- 为什么追问这个
- 当前主题已确认/未决
- 综合报告路径或摘要
- 明确声明：这是中间产物，不是正式计划
- 下一步：用户确认报告后再进入 Phase 2 或 Phase 3

## 8. 禁止事项

- 不要一次抛出跨主题大问卷
- 不要接受"期末考高分""找工作"表面回答就直接进入下一步
- 不要虚构资料或编造内容
- 不要让主 agent 直接调用 WebSearch/WebFetch 替代子 Agent 检索
- 不要提前生成测评题或计划草案
